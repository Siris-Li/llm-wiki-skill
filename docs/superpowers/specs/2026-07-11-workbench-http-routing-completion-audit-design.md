# #158 工作台 HTTP 与路由契约完工审查与收口

Parent: https://github.com/sdyckjq-lab/llm-wiki-skill/issues/158

GitHub Issue: https://github.com/sdyckjq-lab/llm-wiki-skill/issues/190

## Problem Statement

#158 和关联的 12 个子任务已经关闭，对应改动也已合入主线，但“任务关闭”和“改动合并”本身不能证明原设计已经完整落实。

目前的完成证据散落在父任务、子任务、合并说明、设计文档和代码中。12 个子任务的验收清单没有逐项留下证据，GitHub 也没有保存每次改动自动检查通过的记录。设计文档还在部分工作完成后改过一次；如果直接用最新版设计审查，可能出现先做出结果、再修改标准来适配结果的情况。

已有线索还包括：仍有部分入口使用旧规则、若干合并前审查意见没有明确处理、后台启动和请求处理的分离程度可能与原承诺不一致。这些只是待验证线索，不能提前当成结论。

仓库作者不应被迫阅读代码、猜测合并说明或亲自重复测试，才能判断 #158 是否真的完成。项目需要一套独立、可复查的完工审查流程：先固定原始标准和当前结果，完整审查，再修复确认的问题，最后用本地验证、真实页面操作、一次真实模型对话和 GitHub 自动检查共同证明结果。只有所有硬条件满足后，#158 才能关闭。

## Solution

建立一套专门面向 #158 的“完成后审查与关闭流程”。

审查先固定开工时的设计、父任务验收项、12 个子任务及相关产品约定，形成不会被后续修改覆盖的原始标准。后来修改过的设计单独作为设计变更审查：合理改进可以接受，但必须说明原因，并证明没有降低安全、隐私、数据边界、用户主流程或原有承诺。

审查采用双向核对：

- 从原始要求出发，确认每项承诺都有对应结果和实际证据。
- 从当前所有工作台请求入口出发，确认每个入口都符合统一规则，或有明确、合理、可追踪的例外。

审查期间先记录并固定完整结果，不立即修改，以免修复后的状态掩盖原始遗漏。初次审查结论固定后，再处理问题：原子任务没有完成的，重新打开原任务；审查新发现的问题，新建任务并关联 #158。修复完成后重新运行全部相关验证。

验收不仅检查代码，还要实际启动工作台，走完当前知识库、当前对话、wiki 页面、活地图、产出物和消息流等主要操作。人工验收在临时知识库中进行一次小规模真实模型对话；其余成功、失败和边界情况使用隔离环境与模拟数据覆盖，不接触用户真实知识库。

GitHub 必须建立自动检查，覆盖普通检查和虚拟浏览器中的关键操作。自动检查失败时，改动不能进入主线，也不能通过直接写入主线绕过。真实模型不放进线上自动检查，避免费用、密钥和外部网络波动。

审查报告单独保存在仓库中：前半部分用普通人能理解的语言说明结果、影响和是否已解决；后半部分保留完整专业证据，供以后复查。12 个子任务逐项补齐验收记录，并正式建立与 #158 的父子关系。所有硬问题解决、人工验收通过、GitHub 自动检查通过且主线保护生效后，更新并关闭 #158。

## User Stories

1. As a 仓库作者, I want 得到一个明确的完成结论, so that 我不必通过阅读代码来猜测 #158 是否真的落实.
2. As a 没有开发经验的仓库作者, I want 报告开头用简单直白的语言说明结果、影响和处理状态, so that 我能独立理解审查结论.
3. As a 未来复查这项工作的维护者, I want 报告后半部分保留完整专业证据, so that 我能重复验证每个判断.
4. As a 审查者, I want 在审查开始时固定主线版本、父任务和设计基线, so that 审查期间标准或结果不会悄悄变化.
5. As a 审查者, I want 以正式开工前已经定稿的设计作为原始标准, so that 后来修改的文档不能反向证明已有实现正确.
6. As a 设计维护者, I want 后来发生的设计修改被单独列出和判断, so that 合理改进可以保留而要求缩水会被发现.
7. As a 实现者, I want 具体做法可以与原设计不同但必须证明结果相同或更好, so that 审查不会被没有实际价值的形式差异束缚.
8. As a 工作台用户, I want 安全、隐私、数据边界、主要操作和明确承诺不能因实现方式变化而打折, so that 我可以继续放心使用本地知识库.
9. As a 仓库作者, I want 父任务的每条验收项都有对应结果和证据, so that 我知道勾选状态是否可信.
10. As a 审查者, I want 开工时设计中的每项要求都能对应到当前结果, so that 我能发现子任务拆分时遗漏的承诺.
11. As a 审查者, I want 12 个子任务及其合并结果都纳入追踪, so that 我能区分任务描述、完成说明和实际结果.
12. As a 审查者, I want 从当前所有工作台请求入口反向核对设计, so that 我能发现没有被原任务清单点名的旧入口或例外.
13. As a 报告读者, I want 每项要求都被标为已落实、换一种方式落实、部分落实、未落实或暂时无法验证, so that 我能看见真实状态而不是笼统的通过说明.
14. As a 仓库作者, I want 实际运行结果优先于任务关闭状态和合并说明, so that “作者说测试过”不会被直接当成完成证明.
15. As a 审查者, I want 先完成并固定独立审查再开始修复, so that 原始遗漏不会被后续改动掩盖.
16. As a 未来维护者, I want 初次审查记录包含当时的版本、环境和验证结果, so that 我能追溯结论基于什么作出.
17. As a 审查者, I want 早已存在但会影响 #158 目标、安全或主要操作的问题也被纳入, so that “不是这次引入的”不能成为绕过验收的理由.

## Implementation Decisions

- 当前规格编写阶段只新增并发布本文档，不执行审查、不修改工作台、不改变 GitHub 设置，也不更新或关闭已有任务。
- 正式执行时先固定审查对象，包括审查开始时的主线版本、#158、#165 至 #176、对应合并内容、开工时设计、当前产品约定和相关决策记录。
- 权威依据按“当前产品事实与长期决策、开工时设计、父任务验收、子任务验收、合并说明”的顺序使用。合并说明和任务关闭状态只能作为实施方声明，不能单独证明完成。
- 开工前最后定稿的设计作为原始基线。实施期间或实施后修改的设计进入“设计变更表”，逐项记录原要求、新要求、修改原因、影响和是否接受。
- 关键底线严格执行：安全、隐私、三类数据边界、用户主要操作和原始明确承诺不能通过设计变更或替代实现降低。
- 具体实现方式允许变化。只有在用户结果相同或更好、底线不降低、测试能够长期防止退化时，才可判为“换一种方式落实”。
- 每条要求进入同一张证据矩阵，至少记录原始要求、对应子任务、当前结果、自动验证、实际操作验证、结论和遗留处理。
- 证据矩阵使用五种结论：已落实、换一种方式落实、部分落实、未落实、暂时无法验证。任何结论都必须附证据，不能只引用任务勾选或合并说明。
- 审查同时从要求向结果核对、从当前所有请求入口向设计反向核对。真实请求入口、入口登记、前台调用和测试覆盖必须能够相互对应。
- 初次审查是只读阶段。发现先进入固定报告，不边审边改；初次报告完成后才开始问题处理。
- #158 范围包括本次改动和所有会影响其目标、安全、数据边界或主要操作的既有问题。与这些目标无关的小问题不扩入。
- 当前仍使用旧规则的入口逐个判断。原设计明确包含、涉及敏感能力或会破坏统一规则目标的入口必须处理；真正超出范围的辅助入口只有在理由和后续安排明确后才可作为例外。
- 错误提示不要求逐字不变。正常流程和结果必须不退化；错误提示只有在含义更清楚、影响不被掩盖、用户仍知道下一步做什么时才可接受变化。
- 阻止关闭的问题包括：原始明确承诺未完成、安全或隐私风险、三类数据串写、主要操作退化、稳定错误含义错误、关键并发或流式状态错误、核心验证缺失、线上自动检查或主线保护未生效。
- 不阻止关闭的问题限于不影响结果的排版、命名偏好、注释和轻微整理；它们可以另行记录。
- 原子任务承诺没有完成时重新打开原任务。审查中新发现的问题新建任务并关联 #158。每项修复使用独立且大小合适的分支、提交和合并请求，避免把所有问题混在一个大改动中。
- 每项修复完成后先做相关验证，再做完整复验。初次发现、处理决定、修复结果和最终复验必须在报告中分别保留。
- 新增独立审查报告，不覆盖原设计。报告正文用简单直白的语言给出结果、影响、处理状态和关闭判断；专业部分保留完整证据、复现条件、测试结果和判断依据。
- 严重安全问题的完整细节在修复前不得进入公开文档、公开任务、公开分支或公开日志。修复完成后，公开记录影响、修复结果和安全的复验方法。
- GitHub 新增一套稳定的自动检查流程，覆盖确定性检查、构建、测试和虚拟浏览器关键操作。真实模型凭证和真实模型调用不进入线上自动检查。
- 主线启用保护：自动检查是合并前必过条件，直接写入主线不能绕过该条件。保护应在自动检查已合入并成功产生检查记录后启用，避免先锁死主线。
- #165 至 #176 建立正式父子关系，逐项回填验收证据。已关闭但仍带“可开工”含义的错误标签一并清理。
- #158 的最终结论只允许三种：可以关闭；基本完成但仍有阻塞项；存在范围缩水或关键风险、不能关闭。
- 只有全部阻塞项修复、人工验收通过、GitHub 自动检查通过、主线保护生效、遗留审查意见逐项处理、子任务证据补齐后，才更新并关闭 #158。

## Testing Decisions

- 好测试证明用户或系统边界能够观察到的行为，不锁死内部文件名、函数名或某种唯一实现方式。
- 最高测试入口是实际工作台主路径：在隔离数据中启动真实前后台，从页面完成关键操作，并同时观察界面结果、网络结果和磁盘落点。
- 要求追踪本身也是一个测试入口。每项父任务和子任务要求必须有当前主线证据、自动验证和实际操作证据，不能用单一测试替代完整证明。
