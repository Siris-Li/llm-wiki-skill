## 1. 背景与重新定性

#159 在迁移图谱底层能力时发现两类历史问题：

1. 生成端用 Markdown 文件名作为节点 ID。递归扫描时，`wiki/entities/foo.md` 与 `wiki/topics/foo.md` 都会得到 ID `foo`。
2. 引擎为缺失 ID 的畸形输入生成 `node-N` / `edge-N`，可能与真实输入碰撞。
3. 重复 ID 进入引擎后，不同子系统采用“先写胜出”或“后写胜出”，同一个 ID 可能指向不同对象，造成孤儿节点、度数错算和社区统计失真。

这些问题真实存在。但 v2 把“图谱内部用错了身份”直接转化成“要求用户重命名知识文件”，又把改名后的链接风险留给用户人工处理。外部问卷与仓库事实复核后，确认该方向有一个致命反例：

```
改名前：wiki/entities/foo.md + wiki/topics/foo.md
链接：  [[foo]]

若把 topics/foo.md 改成 topics/foo-2.md：
  [[foo]] 不会断链
  [[foo]] 会全部落到仍叫 foo.md 的 entities/foo.md
  其中原本想指 topics/foo.md 的链接会静默指错
```

因此，“不改链接 = 零误改”不成立。A 方案没有消除误指，只是隐式地把所有未改链接判给了保留旧名的页面。

本设计重新把问题拆成三层：

- **页面身份**：图谱怎样唯一识别一个 Markdown 页面。
- **链接解析**：裸名、路径、别名和标题锚点怎样映射到页面。
- **可选改名**：用户确实想改文件名时，怎样预览、确认、更新和撤销。

## 2. 目标与非目标

### 目标

1. 图谱内部不再把文件名当作唯一身份；不同目录的同名页面可以同时存在并正确展示。
2. 裸 `[[foo]]` 只在目标唯一时解析；重名时不猜、不连错，并给出可定位的歧义提示。
3. 明确路径链接能稳定指向目标；别名、跨页锚点、同页锚点、自链接和附件 wikilink 都有不误报的确定规则。
4. 引擎收到旧数据或畸形输入时仍统一去重、稳定降级并报告问题。
5. 工作台和离线 HTML 使用同一套图谱语义；生成图谱与严格检查使用不同退出语义；离线/CI 默认不修改知识库文件。
6. 用户主动改名时，先看到全部影响，再确认修改；改名、确定性链接更新与固定位置迁移可在崩溃后恢复，并且绝不静默覆盖外部编辑器的新内容。
7. 大小写与 Unicode 路径差异在 macOS、Windows 和 Linux 上采用同一套解析与冲突规则；新文件名另受可移植命名规则约束，不依赖宿主文件系统碰巧怎样比较文件名。
8. 保持现有图谱页面范围：只把六类正式 wiki 页面纳入图谱；检查与改名可以扫描更宽的 Markdown 集合，但不得改写原始素材。

### 非目标

- 不做实体解析，不判断两个不同页面是否描述同一个现实对象。
- 不用模型自动猜测歧义裸链接的真实意图。
- 不把 UUID 写入每个 Markdown 页面；相对路径足以解决当前身份碰撞。
- 不把工作台扩成通用 Markdown 编辑器。
- 不在本 issue 内支持普通 Markdown 链接作为图谱边；本次只治理 `[[wikilink]]`。
- 不把图片、PDF 等附件 wikilink 变成图谱节点或图谱边；它们只需要避免被误报成 Markdown 页面断链。

## 4. 依据

### 4.1 仓库内事实

- `scripts/build-graph-data.sh` 当前以 `basename` 生成节点 ID，并递归扫描目录。
- 当前图谱扫描范围是 `entities`、`topics`、`sources`、`comparisons`、`synthesis`、`queries` 六类目录，不等于任意 `wiki/**/*.md`。
- 当前链接解析只去掉 `|别名`，不会去掉 `#标题`，也不会把路径链接解析到 basename 节点。
- 根 `index.md` 会保存页面 wikilink；`raw/` 保存原始素材，改名流程不能把两者当作同一类文件处理。
- 工作台页面列表和读取接口已经使用 `wiki/.../*.md` 相对路径。
- 图谱固定位置已经以 `source_path` 为稳定 key，而不是 basename ID。
- 当前“暂停图谱监听”只会合并重建事件，不会锁定 Markdown 文件，也不会阻止 Obsidian 写入。
- ADR-29 明确规定图谱是 wiki 结构的派生视图；不应为了满足图谱内部键值而强迫修改知识源。
- ADR-32 要求工作台和离线 HTML 共用同一套图谱语义。
- PRODUCT.md 明确工作台不是 Obsidian/Logseq 替代品，不提供通用手写编辑器。

### 4.2 外部产品事实

- Obsidian 支持重命名时更新内部链接；关闭自动更新后会改为询问。
- Obsidian 支持最短唯一路径、相对路径和 vault 内完整路径。
- Obsidian 支持 `[[#同页标题]]`、`[[#^同页块]]` 和带扩展名的附件 wikilink；这些语法不能按普通跨页 Markdown 链接处理。
- Notion 用独立 UUID 区分页面身份与标题，证明“标题不应兼任身份”；但 llm-wiki 保持普通 Markdown 可读性，不直接照搬 UUID。
- Dropbox / Google Drive 的重名体验只能证明“改名前需要用户确认”，不能证明 wikilink 应如何修复。

## 5. 目标数据流

### 5.1 文件发现策略

语法解析共用一套实现，但不同消费者不能共用一个含糊的“扫描所有 Markdown”开关：

| 消费者 | 读取正文的来源集合 | 目标索引 | 写入集合 | 完全不扫描 |
|---|---|---|---|---|
| 图谱节点与边 | `wiki/entities/**/*.md`、`topics`、`sources`、`comparisons`、`synthesis`、`queries` | 知识库内全部 Markdown 与附件路径；Markdown 用于判断裸名是否真的唯一，附件只判断存在性 | 无 | 隐藏/第三方目录、临时目录、生成物；`wiki/overview.md`、未知 `wiki/` 目录、`raw/` 与根元数据只建目标索引，不读正文、不建节点 |
| lint | `wiki/**/*.md` 与根 `index.md` | 知识库内全部 Markdown 与附件路径 | 无 | `raw/` 正文、`.wiki-schema.md`、隐藏/第三方目录、生成物 |
| 改名影响扫描 | 全部 Markdown；其中 `raw/` 与 `.wiki-schema.md` 只读扫描 | 知识库内全部 Markdown 与附件路径 | `wiki/**/*.md`、`index.md`、`log.md`、`purpose.md` | `.obsidian/**`、`.git/**`、`.wiki-tmp/**`、`node_modules/**`、图谱 JSON/HTML |

“知识库内全部 Markdown”只建立路径与 basename 索引，不代表所有页面都成为图谱节点。链接目标存在但不属于六类正式页面时，该链接对 Obsidian 仍有效，图谱不建边，也不报 `broken_wikilink`。`raw/` 中发现的相关链接只进入预览的“不会修改”区域，永不自动改写。

六类正式页面的节点类型由第一个目录段确定；因此实现不需要为 `wiki/notes/` 等未知目录猜节点类型。新增正式目录必须先修改本表、节点类型契约和对应测试。

### 5.2 构建图谱

```
六类正式图谱页面
  ↓ 转为 POSIX 分隔符并做知识库边界校验；保留实际大小写与 Unicode
节点 id = source_path = 实际的 wiki/<目录>/<页面>.md
  ↓ 另建不持久化的 portable_key = Unicode 17.0 NFC(Default_Case_Fold(NFC(path)))
  ↓ 建两个解析索引
exact path / portable path → 0 / 1 / N 个候选
portable basename → 0 / 1 / N 个候选
  ↓ 扫描每个页面的 wikilink
明确路径、精确命中 ───────→ 精确目标 → 建边
明确路径、仅可移植 key 命中 → 唯一目标 → 建边并提示非规范写法
裸名且候选唯一 ─────────→ 唯一目标 → 建边
裸名且候选超过一个 ─────→ 不建边   → ambiguous_wikilink
目标不存在且有“待创建”标记 → 不建边   → pending_wikilink
其他 Markdown 目标不存在 ──→ 不建边   → broken_wikilink
同页锚点或显式自链接 ─────→ 解析成功 → 不建自边、不告警
存在的非图谱 Markdown / 附件 → 解析成功 → 不建边、不报页面断链
  ↓
成功写出 graph-data.json + 紧凑警告详情
  ↓
共享图谱引擎入口兜底去重
  ↓
工作台 / 离线 HTML
```

可降级的数据错误不阻止 `graph-data.json` 生成。只有路径不可读、输出无法原子写入、JSON 无效等系统故障才让构建失败。`check` 只读报告并返回成功；`check --strict` 在报告生成后，若仍有 `error` 才返回数据错误码。工作台和离线导出只调用非严格构建；CI 要阻断时显式调用严格检查。

### 5.3 用户主动改名

```
用户从警告或页面动作进入“解决同名/改名”
  ↓
先扫描：目标名称、已有文件、可编辑与只读引用、固定位置、文件集合和内容摘要
  ↓
展示完整预览
  ├─ 明确路径链接：默认勾选确定性更新
  ├─ 唯一裸链接：按改名前解析结果给出确定性更新
  └─ 歧义裸链接：逐条选择目标，或取消本次改名
  ↓ 用户确认
取得当前知识库的单操作锁；重新扫描文件集合与全部摘要
  ├─ 任一文件新增、删除或变化：预览失效，不做写入
  └─ 完全一致：建立持久操作记录与原内容备份
  ↓
暂停图谱监听（只负责合并重建，不充当文件锁）
  ↓
写入已确认的精确链接更新 + 迁移固定位置 key + 最后改名源 Markdown
  ├─ 大小写/Unicode 等价改名：经同目录唯一临时名中转
  ├─ 进程内失败且没有外部冲突：自动回滚并恢复旧文件名
  ├─ 外部编辑冲突：停止后续写入，保留外部新内容与本次备份，进入恢复界面
  └─ 进程崩溃：下次启动按持久记录继续回滚或确认已提交状态
  ↓ 内容提交成功
标记操作已提交，恢复监听并只触发一次重建
  └─ 重建失败：保留已经正确提交的 Markdown，图谱标为待重建并允许重试
```

持久操作记录放在知识库 `.wiki-tmp/rename-ops/<operation-id>/`，至少包含状态、旧/新相对路径、扫描文件清单、原内容摘要、计划替换区间、原内容备份、写入后摘要和固定位置变更。状态只有 `prepared`、`applying`、`committed`、`rolled_back`、`conflicted`；启动恢复不得猜测未知状态。

改名流程不直接写 `graph-data.json`。所有待写文件先在同一文件系统暂存并完成全量版本复核；每个文件落盘前再次核对原摘要。若某个已由本流程写入的文件随后又被外部程序修改，回滚不得覆盖该新内容，而是保存冲突副本并要求用户选择。这里保证的是“无静默数据丢失且总能恢复”，不是在外部并发写入后强行声称“完全恢复原状”。

## 6. 组件边界

### 6.1 共享 wikilink 解析与路径索引能力

在根目录脚本能力区提供一个单一解析器，作为以下流程的共同事实来源：

- 图谱边生成；
- lint 的断链与歧义检查；
- 工作台改名前影响扫描；
- 可选的确定性链接改写计划。

工作台调用根目录脚本能力，不再用 server 里的 `find + basename + uniq` 重写一套扫描规则。实现必须兼容项目默认的 macOS Bash 3.2，复杂索引与解析放在现有 Node 脚本侧，不依赖 Bash 关联数组。

解析规则：

1. 识别 `[[target]]`、`[[target|display]]`、`[[target#heading]]`、`[[target#^block|display]]`、`[[#heading]]`、`[[#^block]]` 及前缀为 `!` 的嵌入写法。
2. `|display` 只影响显示；`#...` 只影响页内位置；二者都不参与跨页面身份匹配。本文不把 frontmatter 页面别名纳入解析，避免把显示别名与页面身份混为一谈。
3. 扫描到的实际相对路径只统一分隔符、移除安全的 `.` 段并拒绝越界，不静默小写，也不改写其 Unicode 码点；正式图谱页面的该值同时作为节点 `id` 与 `source_path`。
4. 可移植比较 key 固定为 `NFC(Default_Case_Fold_Unicode_17_0(NFC(posix_path)))`。实现使用 Unicode 17.0 `UnicodeData.txt` / `DerivedNormalizationProps.txt` 完成固定 NFC，使用 `CaseFolding.txt` 中状态 C/F 的默认映射并排除 Turkic 专用 T 映射；不能用运行时 `String.normalize`、JavaScript 小写转换或宿主 ICU 数据代替。生产测试必须通过同版本 `NormalizationTest.txt` 的 NFC 一致性向量。
5. `wiki/.../page.md` 和省略 `.md` 的路径先按实际相对路径精确匹配；精确失败后才查可移植 key。后者只有一个候选时可以解析，但产生 `noncanonical_wikilink`；多个候选时按歧义处理。
6. 裸名按知识库全部 Markdown 的可移植 basename key 建索引，仅有一个候选才成功。因此正式图谱页与 `raw/` 中同名 Markdown 同样会让裸链接变成歧义，图谱不能假装它唯一。
7. 两个实际文件的完整路径若映射到同一个可移植 key，产生 `portable_path_collision`。不得合并或覆盖节点；非严格构建仍产出两者，`check --strict` 返回数据错误。
8. `[[#heading]]`、`[[#^block]]` 解析为当前页面；明确链接若最终仍指向来源页，也视为自链接。两者都不建图谱自边、不报断链，可保留在页面内容中。
9. 目标具有非 Markdown 扩展名时按附件处理。附件存在则不建图谱边、也不报 `broken_wikilink`；附件不存在仍由现有附件健康检查处理，不混入页面身份警告。
10. 围栏代码块和行内代码中的示例文本不算真实链接。解析器必须返回语法节点种类，改写器只接受 `page_wikilink`，不能对原文重新做一次全文字符串替换。
11. 精确形态 `[待创建: [[target]]]` 与 `[To create: [[target]]]` 中的缺失目标产生 `pending_wikilink`，严重度为 `warning`；它不是 `broken_wikilink`。目标后来存在时正常解析并建边，lint 另提示清理过期“待创建”标记。其他自然语言前缀不享受豁免，避免任意正文把真正断链降级。
12. 每个真实链接出现位置必须包含：原始链接、来源相对路径、1 起算行列、UTF-8 半开字节区间 `[start_byte, end_byte)`、原文件 SHA-256、解析种类、解析目标或候选集合引用。读取原字节区间必须能精确还原 `raw_link`，否则整个文件的改写计划无效。
13. 同一文件的替换按 `start_byte` 从后向前应用；写入前同时核对文件 SHA-256、文件集合摘要与每个原始字节片段。任何一项不一致都使预览整体失效，不允许部分继续。

### 6.2 可移植文件名与改名目标校验

链接比较 key 和文件名是否合法是两套规则，不能混成一个 case-fold 判断。主动改名只允许在原目录内修改文件名；移动到其他目录不属于本 issue。

新文件名输入可省略末尾 `.md`，保存时只补一个 `.md`。以下任一情况直接拒绝，且在写入前后都校验：

- 空名、`.`、`..`、路径分隔符、NUL 或其他控制字符；
- Windows 非法字符 `< > : " / \\ | ? *`，以及末尾空格或句点；
- 大小写不敏感的 Windows 保留 stem：`CON`、`PRN`、`AUX`、`NUL`、`COM1`–`COM9`、`LPT1`–`LPT9`，无论是否带扩展名；
- 会破坏 Obsidian wikilink 语法的 `#`、`|`、`^`、`[[`、`]]` 或 `%%`；
- 与任一现有路径的可移植 key 碰撞，目标就是当前文件且属于大小写/Unicode 等价改名的情况除外。

中文、空格和一般 Unicode 字符保持允许。大小写或 Unicode 规范形式的等价改名必须先改到同目录内一个经实际占用检查的唯一临时名，再改到目标名；操作记录必须同时保存三段路径，确保 Windows 和默认 macOS 卷上可恢复。

### 6.3 生成端

- 节点 ID 从 basename 改为规范相对路径。
- `id` / `source_path` 保留实际大小写与 Unicode；`portable_key` 只在构建进程内使用，不进入公开 graph-data 契约。
- 边端点、社区成员、学习路径、起点和洞察中的节点引用全部使用路径 ID。
- `source_path` 与 ID 保持同值，先保留字段以维持公开数据契约清晰度。
- 生成端不再因同 basename 丢弃六类正式图谱页面；未知 `wiki/` 目录、根元数据和 `raw/` 不进入节点数组。
- 重名只有在裸链接实际产生歧义时影响关系；同名本身可作为提示，不是构建失败。

### 6.4 引擎层

- `projectGraphInput` 与 `buildAtlasModel` 的 raw 入口共用一次规范化与去重能力。
- 重复节点、边、社区 ID 保留首个并生成警告；所有下游只接收唯一数组。
- 自动节点/边 ID 先收集显式 ID，再循环选择未占用值。
- 引擎把 ID 视为不透明字符串，不从 ID 猜文件路径；页面路径始终优先使用 `source_path`。
- 旧 basename graph-data 仍能打开；新构建才切换为路径 ID。

### 6.5 工作台

- `GraphWarningsBanner` 同时展示重复输入、歧义链接、真正断链、待创建链接、非规范路径写法和可移植路径冲突摘要。
- 图谱成功但含 `error` 时，工作台状态仍为可阅读的 `ready + warnings`；只有构建工具或文件系统故障才进入“图谱重建失败”。
- 警告详情只显示知识库内相对路径，并从紧凑警告组展开完整出现位置；不得把候选集合复制到每一条出现位置。
- “解决”动作进入专用预览，不增加通用 Markdown 编辑器。
- 页面改名端点必须沿用现有本地可信来源检查；知识库根、来源、目标、暂存文件和固定位置文件都经 `realpath`/父目录真实路径校验，拒绝越界与不受控符号链接。
- 同一知识库同时只允许一个改名操作；重复提交同一 `operation_id` 返回原状态，不得再次写入。
- 选择知识库与服务启动时都扫描未完成的改名记录；存在 `applying` 或 `conflicted` 时先进入恢复流程，不允许静默开始下一次改名。
- 改名期间复用现有图谱监听暂停/恢复能力以避免重复重建，但不得把监听暂停描述成文件锁或并发保护。

### 6.6 离线 HTML 与 CLI/CI

- 离线 HTML 嵌入紧凑警告组，显示摘要、候选路径和被忽略的歧义关系，不尝试写文件。
- 图谱构建命令：只要能产生有效降级图谱就返回 `0`；系统/读写/格式故障返回 `1`。工作台和离线导出只调用该命令。
- `check`：只读输出完整报告，数据问题不改变退出码；工具自身失败返回 `1`，与现有 lint 语义一致。
- `check --strict`：报告成功生成且没有 `error` 时返回 `0`；存在 `error` 时返回 `2`；工具自身失败仍返回 `1`。它不生成或修改 Markdown。
- CI 需要阻断时显式调用 `check --strict`；需要保留降级产物时先运行构建，再运行严格检查。两者都永不修改 checkout 中的知识 Markdown。
- 警告数据不得包含用户主目录等本机完整路径。

## 7. 告警契约

```ts
type GraphWarningCode =
  | 'duplicate_node_id'
  | 'duplicate_edge_id'
  | 'duplicate_community_id'
  | 'generated_id_collision'
  | 'ambiguous_wikilink'
  | 'broken_wikilink'
  | 'pending_wikilink'
  | 'noncanonical_wikilink'
  | 'portable_path_collision';

type GraphWarningSeverity = 'error' | 'warning';

interface GraphWarningSummary {
  build_id: string;
  total_groups: number;
  total_occurrences: number;
  error_occurrences: number;
  warning_occurrences: number;
  by_code: Partial<Record<GraphWarningCode, number>>;
  details_ref: 'wiki/graph-warnings.json';
  details_sha256: string;      // 见下文的规范化详情摘要口径
}

interface GraphWarningCandidateSet {
  candidate_set_id: string;
  candidate_count: number;
  candidates: string[];        // 排序后的知识库相对路径，只保存一次
}

interface GraphWarningOccurrence {
  occurrence_id: string;
  source_path: string;         // 仅知识库相对路径
  line: number;                // 1 起算
  column: number;              // 1 起算
  start_byte: number;          // UTF-8 半开字节区间
  end_byte: number;
  raw_link: string;
  file_sha256: string;
  link_kind: 'page_wikilink' | 'same_page_anchor' | 'attachment_wikilink';
  read_only: boolean;
}

interface GraphWarningGroup {
  warning_id: string;
  code: GraphWarningCode;
  severity: GraphWarningSeverity;
  message: string;
  id?: string;
  target_key?: string;
  candidate_set_id?: string;
  occurrence_count: number;
  occurrences: GraphWarningOccurrence[];
}

interface GraphWarningBundle {
  version: 1;
  build_id: string;
  summary: GraphWarningSummary;
  candidate_sets: GraphWarningCandidateSet[];
  groups: GraphWarningGroup[];
}
```

字段命名统一使用现有 graph-data 风格的 `snake_case`。生成端把摘要放进 `graph-data.json.meta.warning_summary`，把完整紧凑详情写入派生文件 `wiki/graph-warnings.json`。两份数据共享 `build_id`，摘要同时保存规范化详情 SHA-256：对 `{ version, build_id, candidate_sets, groups }` 按固定键序紧凑序列化后的 UTF-8 字节计算，不把 `summary.details_sha256` 本身纳入输入，避免文件自摘要形成循环。`graph-data.json` 与侧车内的 `details_sha256` 必须相同，读取端按同一投影重新计算。生成端先在 `.wiki-tmp/` 写完并验证两份临时文件，再先替换警告详情、最后替换 `graph-data.json` 作为提交点；读取时只有 `build_id` 与规范化详情 SHA-256 都匹配才展示详情，否则保留摘要、标记详情不可用并触发重建，不能把两次构建的数据混在一起。

工作台按警告组分页读取详情；CLI 可流式输出完整报告。候选集合按排序后路径数组生成稳定摘要，同一集合只存一次；序列化规模必须与“出现位置总数 + 不同候选集合的成员总数”线性相关，不能变成“每条链接 × 候选数”。

`warning_id` 由 `code + target_key/id + candidate_set_id` 稳定生成；`occurrence_id` 由 `source_path + file_sha256 + start_byte + end_byte + raw_link` 生成。它们用于分页、预览选择和过期检测，不写入 Markdown。

工作台和 CLI 必须能访问完整详情。离线 HTML 的完整警告 JSON 以规范化紧凑 UTF-8 字节做 gzip level 9 计量，压缩后不超过 2 MiB 时全部嵌入；超过 2 MiB 时按稳定的 warning/candidate 排序在预算内嵌入每组最多前 20 个出现位置和每个候选集合最多前 20 条路径，已嵌入的组/集合始终保留 `occurrence_count` / `candidate_count`，全部全局汇总计数始终保留，并设置 `warning_details_truncated: true`。如果仅全量组/集合头本身仍超过预算，则继续按同一稳定顺序省略尾部整组/集合，并保存 `omitted_group_count` 与 `omitted_candidate_set_count`；界面明确提示用户运行 `check` 查看完整报告。截断器本身必须保证最终嵌入警告数据按同一口径不超过 2 MiB；离线截断只影响详情展示，不改变图谱节点、边和总计数。

默认严重度：重复 ID、歧义链接、真正断链和可移植路径冲突为 `error`；`pending_wikilink`、自动 ID 成功避让和仅大小写/Unicode 写法不规范为 `warning`。严重度不改变图谱构建退出码；只有 `check --strict` 根据最终 bundle 中是否存在 `error` 返回 `2`。

## 8. 错误与恢复边界

| 场景 | 行为 | 是否修改源文件 |
|---|---|---|
| 六类正式页面同 basename，无裸链接歧义 | 全部进入图谱，可显示提示 | 否 |
| 裸链接命中多个候选 | 不建边，列出候选 | 否 |
| 明确路径链接存在 | 精确建边 | 否 |
| 明确路径仅以大小写 / Unicode 等价形式命中唯一页面 | 建边，并提示改成实际路径 | 否 |
| 多个文件的完整路径具有相同可移植 key | 不合并；列出全部实际路径；严格检查失败 | 否 |
| “待创建”标记中的目标不存在 | 不建边，报 `pending_wikilink` warning；严格检查不因此失败 | 否 |
| 其他明确路径或裸链接目标不存在 | 不建边，报 `broken_wikilink` error；非严格构建仍成功 | 否 |
| 同页锚点或显式自链接 | 解析成功但不建自边、不告警 | 否 |
| 现有非图谱 Markdown 或附件目标 | 视为有效链接但不建图谱边 | 否 |
| 引擎收到重复 ID | 保留首个并警告 | 否 |
| 用户取消改名 | 保持原状 | 否 |
| 预览后文件集合或任一内容摘要变化 | 预览整体失效，要求重新扫描 | 否 |
| 改名进程内失败，且没有外部冲突 | 根据操作记录回滚已完成写入与固定位置 key | 自动恢复原状 |
| 改名期间外部程序修改尚未写入的文件 | 停止后续写入；安全回滚本流程已经写入且未再变化的文件 | 不覆盖外部新内容 |
| 改名期间外部程序再次修改本流程已写入的文件 | 标记 `conflicted`，保留外部版本、原备份与本流程版本，进入恢复界面 | 需要用户选择，不丢弃任一版本 |
| 应用在改名中崩溃 | 下次启动先读取持久记录，恢复或进入冲突处理；恢复前禁止同库新改名 | 取决于记录状态，不静默继续 |
| Markdown 与固定位置已提交，但图谱重建失败 | 保留提交结果，图谱标为待重建并允许重试 | 是；不因派生视图失败回滚知识源 |
| 离线 HTML / CI 发现歧义 | 展示或报告；只有显式 `check --strict` 返回数据错误，默认不修复 | 否 |
| 系统故障、文件不可读 | 明确失败，不生成伪成功结果 | 否 |

| 工作台 | 数据错误得到 `ready + warnings`，不是重建失败；正确页面读取、完整详情、恢复入口与解决入口在真实浏览器中可用 |
| 离线 HTML | 节点和关系与工作台一致；2 MiB 内完整嵌入，超限时按规则截断详情并保留总数；不含本机完整路径 |
| 首次迁移 | 节点按 `source_path` 映射；有向边按映射后端点和关系类型匹配；社区成员先映射再比较；边重排不产生假变化；固定位置不丢失 |
| CLI / CI | 构建、`check`、`check --strict` 的 0/1/2 退出语义有契约测试；待创建不使严格检查失败，真正断链与歧义会失败；全程不改 Markdown |
| 可移植性 | 固定 Unicode 17.0 NFC 数据与 C/F 映射在 macOS、Windows、Linux 结果一致并通过 17.0 `NormalizationTest.txt`；覆盖特殊 case-fold、NFC/NFD、中文、空格、保留名、非法字符、末尾点/空格和等价改名中转 |
| 性能 | 真实文件读取与完整 Markdown 解析只做一次目标索引与一次来源扫描；警告序列化满足“出现位置 + 不同候选成员”线性上界，并覆盖离线 2 MiB 阈值 |
| 主动改名 | 文件集合变化、逐文件外部编辑、Obsidian 自动更新、双击提交、大小写改名、固定位置迁移、进程崩溃、启动恢复、冲突保全、单次重建和重建失败重试均有端到端测试 |

各阶段只能在对应行通过后标记完成。阶段三的主动改名不是阶段二路径身份上线的前置条件。

## 10. 分阶段实施

### 阶段一：引擎兜底，可独立合并

1. 入口一次去重，统一重复 ID 语义。
2. 自动 ID 改为实际避让已占用 ID。
3. 建立警告摘要、紧凑详情、候选集合去重与 `graph-warnings.json` 数据通路。
4. 工作台先区分“有效图谱含警告”和“系统构建失败”，离线宿主接入同一警告语义。
5. 更新 #159 行为契约与 issue-270 回归测试。

### 阶段二：路径身份与链接解析

1. 落实第 5.1 节三套文件发现策略，建立共享 wikilink 解析、全 Markdown 目标索引与固定 Unicode 17.0 比较能力。
2. 生成端切换到路径 ID，所有节点引用同步切换；只有六类正式页面成为节点。
3. 裸唯一/歧义、明确路径、待创建/真正断链、同页/自链接、附件、非图谱页和代码排除按第 6.1 节处理。
4. 分离图谱构建、`check` 与 `check --strict` 的退出语义。
5. 按 `source_path` 一对一映射旧节点，再同步映射有向边端点和社区成员，避免节点、关系、社区与固定位置的假变化。
6. 工作台与离线 HTML 展示一致摘要；工作台可分页读取完整详情，离线按 2 MiB 上限明确降级。

### 阶段三：用户主动改名

1. 完整影响预览、只读引用提示和跨平台目标名校验。
2. 基于文件摘要与精确字节区间完成确定性更新，歧义逐条确认。
3. 持久操作记录、原内容备份、同库互斥、文件集合复核、固定位置 key 迁移、等价改名中转和启动恢复。
4. 监听暂停只合并重建；内容提交后单次重建，重建失败不回滚正确知识源。
5. 端到端测试覆盖 Obsidian/外部编辑并发、双击提交、进程崩溃、冲突保全和恢复。

阶段三不是阶段二的前置条件。同名页面在阶段二已经能安全进入图谱；改名只是帮助用户改善知识库命名，不再是修复图谱的唯一办法。

## 11. 迁移与兼容

- **现有无重名知识库**：内容和链接不改；重新构建后节点 ID 从 basename 变为相对路径，视觉和关系结果应保持一致。
- **现有有重名知识库**：六类正式同名页面都进入图谱；目标索引还会考虑 `raw/` 与其他 Markdown，因此原裸链接只在整个知识库确实唯一时解析。
- **旧 graph-data.json**：引擎保持可读；工作台下一次重建生成新身份。
- **旧离线 HTML**：作为自包含历史产物不迁移；新导出使用新规则。
- **固定位置**：首次 basename → 路径 ID 时，存储 key 已是页面路径，可继续命中。用户主动改名会改变路径 key，必须在同一持久操作中把 `.wiki-graph-layout.json` 的旧 key 原子迁移到新 key；目标 key 已存在时预览直接阻止执行。
- **首次节点对齐**：旧 `source_path` 只做 POSIX 分隔符统一、安全 `.` 段移除和知识库边界检查，不做 case-fold。它与新 `source_path` 一对一相等时，建立 `旧 node id → 新 path id` 映射。
- **首次关系对齐**：先用节点映射改写旧边端点，并把缺失 `relation_type` 按引擎现有默认“依赖”规范化，再以有向 `(from, to, relation_type)` 作为语义身份匹配新边；`confidence` 是同一关系的属性，不参与身份。生产生成端维持“每个有向端点对只保留一条关系”的既有约束；遗留重复语义边按稳定输入顺序一对一匹配并告警，不能任意覆盖。
- **首次社区对齐**：先把旧社区成员全部映射为新路径 ID，再运行现有成员重合度匹配。不能直接拿 basename 成员与路径成员计算，否则未变化社区会被误报为全新社区。
- **首次重建结果**：匹配成功的节点、边和社区在 diff/动画中使用新 ID。可复跑验证中，节点误报从 4 增 4 删降为 0，边从 2 增 2 删降为 0，未变化社区重合度从 0 恢复为 1。
- **迁移歧义**：缺失 `source_path`、同一路径出现多个节点或无法一对一匹配时，只回退到完全相同的 ID；不得用 basename 猜。未匹配项按真实新增/删除展示并产生迁移警告。
- **迁移写入边界**：对齐只作用于刷新 diff、动画和固定位置查找，不回写旧 `graph-data.json`，也不修改 Markdown。
- **大小写与 Unicode**：节点继续使用实际相对路径；可移植 key 碰撞只告警和阻止严格检查，不静默合并、改名或覆盖页面。
- **Obsidian**：保留裸 wikilink 兼容，并接受明确路径；不改 `.obsidian/`。Obsidian 自动更新链接被视为外部并发写入，按摘要冲突规则处理，不能假设暂停图谱监听会暂停 Obsidian。

## 12. 风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 路径 ID 触发大量旧快照变化 | 高 | 单独迁移提交；按 `source_path` 对照旧新结果；逐段复核 baseline |
| 首次重建被误判为全库节点、边或社区变化 | 高 | 先映射节点身份，再改写旧边端点与社区成员后比较；方法已验证，生产实现必须通过第 9.2 节验收 |
| 解析器改写误碰代码示例或同一行相同文本 | 高 | 共享解析器排除代码，输出 SHA-256 与精确 UTF-8 字节区间；只按语法节点从后向前替换 |
| 裸链接歧义导致边减少 | 中 | 这是诚实降级；界面明确列候选和解决入口，不静默猜测 |
| 工作台与 Skill 各写一套解析规则 | 高 | 解析能力归根目录脚本，工作台调用而不复制 |
| 数据错误让工作台收不到降级图谱 | 高 | 构建对可降级问题返回成功；只有独立 `check --strict` 因数据错误返回 `2` |
| 改名中途失败或崩溃留下半套结果 | 高 | 持久操作记录、原内容备份、同库互斥、启动恢复与明确状态机 |
| Obsidian 在改名期间同时更新链接 | 高 | 全量摘要复核；不覆盖外部新内容；冲突时保留三份版本并进入恢复界面 |
| 改名后固定位置消失 | 高 | `.wiki-graph-layout.json` 旧 key → 新 key 与 Markdown 修改属于同一操作 |
| 路径或警告泄露本机目录 | 中 | 所有持久化与离线数据只使用知识库相对路径 |
| 大小写、Unicode 或 Windows 文件名在换系统后失效 | 高 | 实际路径作为身份；固定 Unicode 17.0 NFC / case-fold 数据而不依赖 ICU；独立文件名规则拒绝保留名、非法字符与末尾点/空格 |
| 警告候选在每个链接中重复导致产物膨胀 | 高 | 候选集合去重、完整详情侧车、工作台分页、离线 2 MiB 明确截断规则 |
| 大型知识库扫描变慢 | 中 | 一次目标索引、一次来源扫描；性能验收同时限制警告序列化规模 |

## 13. 审查记录

### v1 → v2

- 引擎从多处分别防御改为入口一次去重。
- 自动 ID 从普通前缀改成双下划线前缀。
- 增加生成端与引擎 warnings 通路。
- 新增“强制改名 + 事后链接提示”方案。

### v2 → v3

- 发现 A 的核心反例：改名后裸链接不会断，而会静默落到保留旧名的页面。
- 纠正 Obsidian 事实：它支持重命名时自动更新或询问，并支持路径链接。
- 删除强制改名主线，改为路径身份、歧义不猜、改名可选。
- 删除离线/CI 自动加后缀；所有无界面流程默认只读。
- 删除手工修改 `graph-data.json`；图谱只从 Markdown 重建。
- 修正“自动前缀物理不碰撞”的错误说法，改为实际避让。
- 统一 warning 字段为 `source_path`，禁止完整本机路径进入离线 HTML。
- 增加 macOS Bash 3.2、三重同名、已有后缀、外部并发修改、固定位置和首次迁移验证。
- 完成路径身份可行性验证：共享引擎、Sigma、工作台读取、离线 HTML 与只读解析路线均通过。
- 发现首次迁移会误报全库新增/删除，补充 `source_path` 一对一对齐规则。
- 发现默认 macOS 卷不能可靠共存大小写或 Unicode 规范等价文件，补充独立于文件系统的可移植比较规则。
- 把“方案可行性验证”与“生产实现验收”拆开，避免把原型成功误写成功能已完成。

### v3 全量审查修订

- 把现有 `[待创建: [[...]]]` 从真正断链中分离，避免正常 ingest 天然无法通过严格检查。
- 把“成功生成降级图谱”与“严格检查失败”拆成不同命令和退出语义，保证工作台能拿到警告。
- 为图谱、lint、改名分别固定文件发现与写入边界；六类正式页面之外不再猜节点类型，`raw/` 永不改写。
- 链接出现位置增加原文件 SHA-256 与 UTF-8 精确字节区间，消除同一行相同文本和代码示例误改。
- 改名改用持久操作记录、同库互斥、原内容备份、启动恢复和外部冲突保全；不再承诺覆盖外部新内容来“恢复原状”。
- 把固定位置旧路径 key → 新路径 key 纳入同一次改名操作。
- 将 D12 从“已验证”降为“方向已验证”；固定 Unicode 17.0 NFC 与默认 case-fold，并另立 Windows/Obsidian 文件名规则。
- 明确定义同页锚点、自链接、附件和非图谱 Markdown 目标的行为。
- 保留可复跑方向程序，新增共享引擎到 Graphology 的正式路径 ID 测试；未保留截图的离线人工观察不再算可复跑证据。
- 首次迁移补齐有向边语义 key 与社区成员映射，验证节点、边和社区都不会产生假变化。
- 警告改为摘要 + 完整侧车 + 候选集合去重；工作台分页，离线 HTML 使用明确的 2 MiB 详情上限。
- 明确 `details_sha256` 对排除自身字段的规范化详情投影计算，避免侧车要求包含自身完整文件摘要的循环定义。
- 固定离线 2 MiB 预算按规范化紧凑 JSON 的 gzip level 9 结果计量，避免不同实现各自解释“压缩后”。
- 补齐“仅全量组头也超限”的极端降级：保留全局总数和省略数量，稳定省略尾部详情组，确保 2 MiB 是硬上限而非通常情况目标。
- D12 同时固定 Unicode 17.0 NFC 数据与默认 case-fold，加入官方规范化一致性向量，消除最低 Node 版本所带 ICU 数据可能较旧的隐患。
- 仓库只忽略 `docs/superpowers/plans/`，继续允许 `specs/` 作为意图资产入库。
