# 社区视图 Phase 2 设计：局部地图体验

> 状态：设计稿，已完成工程审查，待拆实施计划
> 日期：2026-07-02
> 关联：PR #94、Issue #93、Issue #95

## 1. 一句话目标

Phase 2 的目标不是马上把社区也迁到 Sigma，而是先做到：

**底层仍然是“全局 Sigma / 社区 DOM”，但用户感觉是在同一张地图里看远景和近景。**

全局图看起来已经有地图感。现在的问题是进入社区后，画面观感明显下降：节点像重新散开，标签拥挤，边线变乱，核心不够稳。Phase 2 要修的是这个整体体验，而不只是“节点位置不同步”这一条。

## 2. 背景

Phase 1 已经完成：

- 全局 Sigma 和社区 DOM 的颜色、底色、字体、节点光晕更统一。
- 社区宽屏形状畸变已修复。
- 社区动画期间的重排卡顿已缓解。

但 Phase 1 只解决了“看起来不是两套皮肤”的问题，还没有解决“进入社区后不好看”的问题。

当前用户实际感受是：

- 全局视角下，社区分布和地图感还可以。
- 进入 DOM 社区后，节点位置、疏密、标签、边线层次一起变差。
- 这不是单一 bug，而是社区近景模式的体验问题。

因此，Issue #93 的范围需要扩大：它不再只是“全局 Sigma spotlight 与 DOM 社区节点布局不同步”，而是“进入社区后局部地图体验下降”。

## 3. 与远期 Sigma 社区迁移的关系

已新建远期 issue：

- #95：远期评估“社区视图迁移到 Sigma，统一全局与社区图谱架构”

这件事不进入当前 Phase 2。

原因：

- 把社区也迁到 Sigma 是更大的架构工程。
- 社区近景不只是画点，还包含标签阅读、边线层次、抽屉联动、节点操作、移动端可读性。
- 如果直接迁移，很容易把这轮“修观感”的目标变成长期重构。

Phase 2 只吸收 #95 的产品原则：**社区应该像全局地图的局部放大版。**

## 4. Phase 2 不做什么

本阶段明确不做：

- 不把社区视图迁移到 Sigma。
- 不把“镜头推进动画”当作第一目标。
- 不追求全局和社区像素级一致。
- 不让所有节点标签同时显示。
- 不重做右抽屉的大结构。
- 不引入新的图谱引擎依赖。

- 不做进入社区的过渡动画。动画移动到 Phase 2.1，等静止落点画面通过验收后再评估。

落点画面不好看时，再顺的动画也没有意义。Phase 2 先把社区页的静止近景做稳、做清楚。

## 5. 体验原则

### 5.1 同一张地图

用户在全局 Sigma 里点击社区后，镜头已经会移动到那片社区。Phase 2 不是要求点击“进入社区”后再做一次更强的靠近，而是要求 DOM 社区页**接住这个已经靠近的区域**：

- 用户刚刚在全局看到的是哪片社区，进入后仍然应像在看同一片。
- DOM 不应重新把节点散成另一种形状。
- “进入社区”更像从远景图层切到近景阅读图层，而不是跳到另一张图。

这要求社区页遵守全局地图的基本空间关系：

- 核心节点大致仍在熟悉的位置。
- 节点之间的相对关系不要突然改变。
- 社区颜色、纸面、节点语言和全局保持一致。
- 返回全局时，Sigma 继续保留刚才社区的选中 / 高亮上下文，让用户知道自己刚才看的是哪一片。

这里的“同一张地图”不是复用 Sigma 当前屏幕上的像素位置，也不是让 DOM 社区重新跑一套布局。它指的是：全局和社区都从同一套图谱规则拿节点基础位置、节点层级、标签集合和边线层级，再分别画到 Sigma 或 DOM 上。

### 5.2 局部近景比远景更清楚

社区页不是简单放大全局图。近景应该更适合阅读：

- 核心节点更突出。
- 普通节点退后。
- 标签更少、更准、更不遮挡。
- 边线分层，不能织成一张网。
- 右抽屉解释社区，地图负责空间关系。

### 5.3 稳定优先于活泼

DOM 社区当前会运行模拟，节点会继续移动。这会让社区页和全局高亮看起来不像同一片地图。

Phase 2 倾向于让社区页默认更稳定：

- 进入社区时优先使用全局同源的节点位置。
- 默认不让模拟持续把节点推到另一种形状。
- 如果需要轻微整理，也应被限制在很小范围内，不能破坏用户刚才在全局看到的形状。

如果某个社区因为稳定位置而显得拥挤，优先降低标签和弱边的存在感，而不是放开节点重新乱跑。节点避让只能作为兜底，而且必须有明确上限。

## 6. 设计方案

### 6.1 社区页改成“局部地图模式”

进入社区后，DOM 社区页仍然保留，但视觉目标从“节点关系图”改成“局部地图”。

局部地图应该具备：

- 同一社区的柔和底色区域。
- 核心节点为第一视觉焦点。
- 重要节点显示标签。
- 普通节点只显示点，必要时 hover 再显示标签。
- 社区内部主关系清楚，弱关系和外围关系降低存在感。
- 右抽屉保留当前社区说明、统计、核心节点和动作。

### 6.2 共享地图规则，而不是共享同一个 renderer

Phase 2 保留两套底层：

- 全局仍由 Sigma 负责。
- 社区仍由 DOM 负责。

但两套底层不应各自发明一套地图规则。它们应共享同一套“社区局部地图规则”：

- 同一批基础节点位置。
- 同一套核心 / 相关 / 外围分层。
- 同一套标签预算。
- 同一套边线重要性分层。

也就是说，本阶段追求的是**一套地图规则，两套画布承载**。这样比“完全迁移社区到 Sigma”小得多，也比“全局和社区各写一套逻辑”更容易保持同步。

如果说“两个视角都走一套代码”，Phase 2 应优先做到的是：位置、层级、标签、边线这些判断逻辑共用一套代码；真正把社区也画在 Sigma 里，则属于 #95 的远期工程。

工程边界需要写死：

- 统一规则层由图谱引擎负责计算。
- Sigma 和 DOM 只消费这份结果，不再各自二次发明核心节点、标签显示、边线强弱和节点位置。
- 若 DOM 为了阅读做微调，微调结果必须能回溯到统一规则层的基础位置，不能变成新的布局来源。
- 后续测试要直接检查：同一份输入下，Sigma 和 DOM 拿到的节点层级、标签集合、边线层级是一致的。

### 6.3 节点位置：默认保持全局同源

社区页进入时，节点位置优先沿用全局图使用的基础位置。

这里的“全局同源位置”定义为：

- 不是 Sigma 镜头移动后的屏幕坐标。
- 不是社区 DOM 自己模拟后的新坐标。
- 是图谱引擎给每个节点算出的基础地图坐标，包括用户固定过的位置。

DOM 社区页可以把这批坐标裁剪、缩放成近景，但不能把它们重新洗牌。

不再让社区页一进入就明显改形状。若有整理逻辑，必须满足：

- 核心节点不漂移太远。
- 社区整体形状不翻转、不旋转、不压扁。
- 普通节点可以轻微避让，但不能把社区变成另一张图。

建议 Phase 2 先采用保守做法：

**进入社区后，节点位置先稳住。**

这里的“稳住”不是说页面不能操作，而是说节点不要继续自动大幅漂移。用户仍然可以点击节点、hover 看关系、打开抽屉、返回全局。

如果少数社区保持原位置后太拥挤，再做小幅避让：

- 核心节点默认不移动。
- 普通节点确实重叠时，可以挪开一点，但建议上限不超过社区可视区域短边的 4%，且不能累计越挪越远。
- 标签挡住核心信息时，可以微调标签位置，优先移动标签而不是移动节点。
- 边线太密时，可以降低弱关系存在感。

但这些整理都不能把社区变成另一种形状。默认原则是：先保留用户刚才在全局里看到的那片结构，只有确实影响阅读时才轻轻整理。

如果轻微整理仍然不够，降级顺序是：

1. 进一步减少普通节点标签。
2. 只保留骨架边和当前交互相关边。
3. 显示“该社区较密集”的轻提示。
4. 仍不放开自由模拟。

### 6.4 标签：从“全显示”改为“预算显示”

社区页最影响观感的是标签拥挤。

Phase 2 应建立标签预算：

- 永远显示：当前社区标题、核心节点、选中节点、搜索命中节点。
- 优先显示：少量高重要性节点。
- 暂不显示：普通外围节点，hover 或选中时再显示。
- 长标题默认截断，不让标签横跨整张图。

目标不是少信息，而是让用户第一眼看懂结构。

预算需要有硬边界，避免实现时又回到“全显示”：

- 桌面默认标签数量控制在 6-10 个左右，随社区大小略微变化。
- 移动端默认标签数量控制在 3-6 个左右。
- 核心、选中、搜索命中优先占用预算；预算不足时，普通节点不显示常驻标签。
- 长标题默认单行截断，完整标题通过 hover、选中或抽屉查看。

### 6.5 边线：从“全部关系同等出现”改为“分层”

社区内关系多时，边线容易变成一团网。

Phase 2 应按重要性分层：

- 核心节点相关边更清楚。
- 当前 hover 或选中节点的一度关系更清楚。
- 二度关系变淡。
- 低权重或外围关系更淡。
- 跨社区或背景关系可以作为淡淡的上下文，不抢视觉。

这样用户能先看出骨架，再按需探索细节。

边线重要性也应来自统一规则层，而不是 DOM 和 Sigma 各自判断。优先信号包括：是否连接核心节点、是否属于骨架边、是否与当前 hover / 选中节点一度相关、关系权重和关系类型。弱边可以存在，但不能盖过主边。

### 6.6 节点层次：核心、相关、外围

社区页节点分三层：

- 核心节点：更大、更深、更稳定。
- 相关节点：中等大小，保留标签机会。
- 外围节点：更小、更淡，减少噪音。

这比单纯按类型上色更适合“看懂这个社区”。

类型信息仍可通过 hover 光晕、图例或抽屉体现，不需要在默认状态抢主视觉。

分层来源要复用已有重要性信号：核心节点、重要节点、起点节点、搜索命中、选中节点、节点权重和关系连接。不要为了社区 DOM 单独写一套“看起来像核心”的规则。

### 6.7 进入过渡：移到 Phase 2.1

Phase 2 不做进入社区的过渡动画。

原因：

- 当前核心问题是社区页落点不好看，而不是动画不顺。
- 动画可能掩盖落点问题，让验收变得模糊。
- 动画会增加浏览器验证成本，分散 Phase 2 的注意力。

Phase 2.1 可以在静止画面验收通过后再做轻量过渡。那时动画只服务于连续感，不改变节点最终位置。

### 6.8 返回全局

从社区返回全局时：

- 回到 Sigma 全局图。
- 保留刚才进入社区时的来源社区上下文。
- Sigma 继续高亮这片社区。
- 用户能看出自己刚才从哪一片回来的。

这比简单重置到全图更有连续感。

来源社区上下文的清除时机也要明确：

- 用户选择另一个社区时，更新为新的社区。
- 用户明确清空选择、重置视图、切换知识库时，清掉。
- 图谱数据刷新后该社区不存在时，清掉。
- 用户只是在社区内 hover、选中节点、搜索当前社区内容时，不应立刻丢失来源社区。

## 7. 预期交互流程

1. 用户在全局 Sigma 图上点击社区。
2. 全局图高亮该社区，右抽屉展示社区摘要。
3. 用户点击“进入社区”。
4. 系统记录来源社区。
5. DOM 社区页打开，但画面像刚才那片社区的近景。
6. 社区页默认只显示关键标签和清晰骨架。
7. 用户 hover 或选中节点时，局部关系增强。
8. 用户返回全局，Sigma 仍高亮刚才那片社区。

## 8. 验收标准

Phase 2 完成后，应满足：

- 进入社区后，不再有“换了一张图”的突兀感。
- 社区节点不会明显散成另一种形状。
- 核心节点第一眼可见。
- 标签不会互相严重遮挡。
- 边线不会织成一团网。
- 右抽屉和地图职责清楚：抽屉讲内容，地图讲关系。
- 全局进入社区，再返回全局，用户能保持空间方向感。

对 #93 的验收也按这个扩大后的标准判断。

可检查标准：

- 同一输入下，全局 Sigma 和社区 DOM 使用同一份节点层级、标签集合和边线层级。
- DOM 社区页使用统一基础坐标生成近景，不启动会把节点推成另一种形状的自由模拟。
- 核心节点在社区页首屏可见，且相对位置不翻转、不旋转、不明显重排。
- 若启用节点避让，核心节点不动，普通节点移动不超过设计上限。
- 常驻标签数量受预算控制，长标题不横跨整张图。
- hover / 选中节点时，一度关系增强，但弱关系不会突然抢占主视觉。
- 返回全局后，来源社区仍处于高亮或选中上下文。
- 桌面和移动端截图都不能出现空白图、标签大片遮挡、弱边盖过主边。

## 9. 验证方式

实现后需要验证三类场景：

### 9.1 真实知识库手动验证

使用 `AI学习知识库`：

- 进入 “AI Agent 对比分析” 这类节点较多社区。
- 对比全局高亮和社区近景。
- 检查节点形状、标签拥挤、边线层次、核心节点可见性。

同时需要覆盖几类固定样本，避免只调好一个真实社区：

- 较密集社区。
- 很小社区。
- 长标题社区。
- 边特别多的社区。
- 没有明显核心节点的社区。
- 断开或弱连接的社区。

### 9.2 自动化浏览器回归

补一个稳定场景：

- 打开图谱。
- 点击一个社区。
- 进入社区。
- 检查社区页仍是局部地图状态。
- 返回全局。
- 检查路线和状态没有丢。

浏览器验证至少覆盖：

- 桌面视口：建议 1440x900。
- 移动视口：建议 390x844。
- 页面不能空白。
- 社区 DOM 根节点处于局部地图状态。
- 核心节点和预算标签存在。
- 返回全局后，来源社区仍高亮。
- 保存关键截图，便于人工复核观感。

### 9.3 引擎单测

覆盖：

- 同一输入下，统一规则层输出稳定。
- 社区阅读态的位置稳定规则。
- Sigma 和 DOM 消费到的节点层级、标签集合、边线层级一致。
- 标签预算规则。
- 边线分层规则。
- 核心、相关、外围节点的层级计算。
- 来源社区上下文的保留和清除规则。

## 10. 风险

### 10.1 节点位置稳住后可能显得拥挤

全局位置适合远景，不一定天然适合近景。

缓解：

- 先用标签预算和边线分层减少噪音。
- 只允许小幅避让和标签微调，不允许把社区形状改掉。
- 如果仍然拥挤，进入降级策略，而不是放开自由模拟。

### 10.2 两套底层仍可能有细微不连续

全局是 Sigma，社区是 DOM，不可能完全一样。

缓解：

- 验收看用户感受，不追求像素级一致。
- 先解决明显体验断裂。
- 把完整迁移 Sigma 留到 #95。
- 加测试防止两套画布的规则输出再次分叉。

### 10.3 标签预算可能藏掉用户想看的节点

标签少了，可能有人觉得信息少。

缓解：

- hover、选中、搜索时补充显示。
- 右抽屉保留核心节点列表。
- 搜索命中永远优先显示。

### 10.4 状态同步可能出错

风险：

- 地图、右抽屉、搜索、返回全局各自维护状态，容易出现地图高亮和抽屉内容不一致。

缓解：

- 明确来源社区上下文。
- 明确清除时机。
- 浏览器回归覆盖进入社区、节点交互、返回全局。

### 10.5 密集社区可能带来性能压力

风险：

- 标签预算、边线分层、避让如果在 hover 或 resize 时频繁重算，密集社区会卡顿。

缓解：

- 基础位置、节点层级、默认标签集合和默认边线层级由统一规则层一次计算。
- hover / 选中只做局部关系增强，不重跑完整布局。
- 如果需要避让，先限制在进入社区时的一次轻量整理。

## 11. 后续分工

Phase 2 后续应拆成实施计划，建议顺序：

1. 先做社区局部地图规则层：位置稳定、节点层次、标签预算、边线分层。
2. 再做 DOM 社区页视觉落地。
3. 再补全局进入社区和返回全局的来源社区上下文。
4. 再补引擎单测和浏览器回归。
5. Phase 2.1 再评估轻量过渡动画。

远期若 Phase 2 仍无法让体验足够连续，再启动 #95 的 Sigma 社区迁移调研。

## 12. 工程审查补充

### 12.1 已确认的方向

- 保留两套画布：全局 Sigma，社区 DOM。
- 共享一套地图规则：位置、层级、标签、边线由图谱引擎统一计算。
- 社区阅读态默认稳住节点，不继续自由自动排布。
- 返回全局时保留来源社区上下文。
- Phase 2 不做过渡动画，动画后移到 Phase 2.1。
- 验收必须包含浏览器级截图和交互验证。

### 12.2 已存在的基础

- 图谱引擎已有统一渲染模型，可输出节点位置、标签可见性、核心节点、边线强弱等信息。
- Sigma 全局路线已经消费统一模型中的节点位置和标签可见性。
- DOM 社区页已经有局部地图样式基础，包括社区底色、节点点状显示、关系聚焦和标签隐藏能力。
- 现有浏览器回归已经覆盖图谱交互、Sigma 全局路线、社区节点地图等场景，可以在其上新增 Phase 2 回归。

### 12.3 不进入本阶段

- 社区视图迁移到 Sigma：留给 #95。
- 进入社区过渡动画：留给 Phase 2.1。
- 右抽屉大改版：本阶段只保证状态联动。
- 新增图谱引擎依赖：继续复用现有引擎。
- 追求像素级一致：本阶段追求空间关系和阅读体验一致。

### 12.6 主要失败模式

- 节点仍被自由模拟推开：用户进入社区后仍觉得换了一张图。需要单测和浏览器截图覆盖。
- 标签预算失效：普通节点标签铺满画面。需要预算单测和截图检查。
- 弱边过强：边线织成网，核心关系看不清。需要边线层级单测和视觉检查。
- 返回全局丢失来源社区：用户不知道自己从哪片回来。需要交互回归覆盖。
- 密集社区无法阅读：需要降级策略，而不是放开节点自由重排。

### 12.7 实施任务

- [ ] T1：统一规则层补强。明确位置、节点层级、标签预算、边线层级的唯一来源。
- [ ] T2：社区阅读态稳住节点。默认关闭或限制会改变社区形状的自动排布。
- [ ] T3：DOM 社区页消费统一规则。视觉只表达规则结果，不再单独发明层级。
- [ ] T4：来源社区上下文。进入社区时记录，返回全局时恢复高亮，明确清除时机。
- [ ] T5：测试与验收。补引擎单测、浏览器回归、桌面和移动截图。
