# 活地图阅读流程内部收拢

GitHub Issue: https://github.com/sdyckjq-lab/llm-wiki-skill/issues/146

## Problem Statement

活地图已经能支持全局图谱、社区阅读、右抽屉、选区和对话联动，但这些动作的编排还散在工作台外壳里。用户现在看到的界面可以继续保持不变，但后续每次想改善图谱阅读体验时，都容易同时牵动图谱画布、右抽屉和对话入口，风险偏高。

本规格的目标不是新增可见功能，而是先把“用户在活地图里阅读和提问”的背后流程收拢成一个清楚的工作台侧流程。这样下一步再做用户可见的图谱阅读体验改善时，不会在零散事件处理里硬改。

## Solution

先做一次纯内部收拢：把活地图阅读流程中“点选、打开摘要、进入社区、打开节点阅读、关闭右抽屉、从选区发起提问、刷新后保留或清理状态”的编排从工作台外壳中集中出来。

用户看到的活地图、右抽屉、按钮、文案和对话发送行为应尽量完全不变。重构完成后，工作台外壳只负责连接活地图、右抽屉和对话入口；活地图阅读流程本身负责决定下一步应该打开什么抽屉、下发什么图谱命令、什么时候清理选区、什么时候切回对话。

## User Stories

1. As a 工作台用户, I want 点选一个图谱节点后仍看到同样的节点摘要, so that 我不会感觉这次整理改变了活地图的基本用法.
2. As a 工作台用户, I want 点选一个社区后仍看到同样的社区摘要, so that 我可以继续从社区入口理解一组页面.
3. As a 工作台用户, I want 手动多选节点后仍看到同样的选区抽屉, so that 我可以把一组页面带入对话.
4. As a 工作台用户, I want 手动单选节点时仍保留选区语义, so that 我不会把临时选区误解成节点详情阅读.
5. As a 工作台用户, I want 从社区摘要进入社区阅读时仍保持当前空间连续性, so that 我知道自己是从全局图谱进入了局部阅读.
6. As a 工作台用户, I want 进入社区时右抽屉仍按原有方式退场或关闭, so that 画布和抽屉之间不会突然跳变.
7. As a 使用减少动态效果的用户, I want 进入社区时继续跳过不必要的退场动画, so that 系统尊重我的动效偏好.
8. As a 工作台用户, I want 在社区摘要中点核心节点时仍先看到节点摘要, so that 我能在进入完整阅读前快速判断页面价值.
9. As a 工作台用户, I want 从社区摘要点“打开详情”后仍能打开节点阅读, so that 我可以进入具体 wiki 页面内容.
10. As a 工作台用户, I want 从节点摘要点“进入所属社区”后仍进入对应社区并选中该节点, so that 我能从单个页面回到它所在的局部结构.
11. As a 工作台用户, I want 在节点阅读中点击“找相关页面”后仍选中邻居页面, so that 我可以继续探索这个节点附近的内容.
12. As a 工作台用户, I want 在节点阅读中点击提问动作后仍把该节点内容带回对话, so that 我可以基于当前页面继续问 agent.
13. As a 工作台用户, I want 在选区抽屉中输入补充说明后仍能随选区一起发问, so that 我的具体问题不会丢失.
14. As a 工作台用户, I want 在社区摘要中输入补充说明后仍能随社区一起发问, so that 我可以围绕整组页面提问.
15. As a 工作台用户, I want 从选区发问后仍关闭右抽屉并清理图谱选区, so that 我清楚自己已经回到对话流程.
16. As a 工作台用户, I want 从社区摘要发问后仍关闭右抽屉并清理图谱选区, so that 不会留下过期的视觉选择.
17. As a 工作台用户, I want 选择“新对话中提问”时仍创建新对话, so that 我可以把图谱探索结果带到新的工作线程里.
18. As a 工作台用户, I want 从活地图发起提问后仍切回对话主区, so that 我能立刻看到即将发送的内容.
19. As a 工作台用户, I want 活地图刷新后当前节点阅读能继续保留, so that 文件变化不会无故打断阅读.
20. As a 工作台用户, I want 活地图刷新后已经消失的节点阅读能安全关闭, so that 我不会读到已经不存在的页面.
21. As a 工作台用户, I want 活地图刷新后离开当前社区范围的节点阅读能清理焦点, so that 图谱状态不会指向错误位置.
22. As a 工作台用户, I want 筛选或搜索隐藏当前阅读节点时仍看到清楚的隐藏提示, so that 我知道页面还在，只是被当前视图规则藏起来.
23. As a 工作台用户, I want 清除图谱选择时只关闭图谱相关抽屉, so that wiki 页面阅读或产出预览不会被误关.
24. As a 工作台用户, I want 关闭右抽屉时图谱选择仍按原规则清理, so that 图谱和抽屉状态不会分离.
25. As a 工作台用户, I want 从节点摘要返回社区摘要时仍保留原来的社区上下文, so that 我可以在社区和节点之间来回看.
26. As a 工作台用户, I want 临时显示被筛选对象后仍能清理临时对象, so that 图谱不会留下过期强调.
27. As a 工作台用户, I want 预览节点时仍只做临时预览, so that 不会误打开抽屉或改变当前选择.
28. As a 工作台用户, I want 固定或取消固定节点位置时仍只影响图谱位置, so that 它不会改变右抽屉内容或知识库内容.
29. As a 工作台用户, I want 图谱重置视图后仍能从节点阅读回到节点摘要, so that 我不会被困在过期阅读状态里.
30. As a 工作台用户, I want 图谱空状态、加载状态和错误状态不受这次整理影响, so that 基本反馈仍稳定.
31. As a 维护者, I want 活地图阅读流程有一个清楚的工作台侧入口, so that 后续改善体验时不用继续往工作台外壳里堆逻辑.
32. As a 维护者, I want 图谱画布继续只负责图谱引擎适配, so that 图谱渲染和工作台阅读流程不会混在一起.
33. As a 维护者, I want 右抽屉继续只负责展示和按钮回调, so that 展示组件不会拥有跨图谱和对话的流程规则.
34. As a 维护者, I want 对话发送流程不被这次改动触碰, so that Prompt Run 重构可以按后续计划单独推进.
35. As a 维护者, I want 活地图阅读流程的测试覆盖完整用户路径, so that 内部收拢不会靠“看起来没报错”过关.
36. As a 维护者, I want 这次不改变知识库清单规则, so that 文件扫描和页面归类可以放到最后一个专项里处理.
37. As a 维护者, I want 用户可见的图谱阅读体验改善被单独排在后续, so that 这次内部收拢不会混入产品改动.
38. As a 未来实现者, I want 文档明确哪些体验改善延后, so that 我不会把“顺手优化”塞进当前重构.
39. As a 未来实现者, I want 回归测试先证明旧行为不变, so that 后续体验改善可以建立在稳定地基上.
40. As a 未来实现者, I want 规格使用“活地图阅读流程”这个统一名字, so that 团队不会把它误叫成图谱组件或图谱动画.

## Implementation Decisions

- 当前推进顺序定为：先做活地图阅读流程内部收拢，再做用户可见的图谱阅读体验改善，再做对话发送流程，最后做知识库清单规则。
- 本次只做内部收拢，用户界面尽量完全不变。
- 本次不改图谱引擎的语义、布局、渲染、动画或图谱数据生成。
- 本次不改右抽屉的视觉设计、文案、按钮布局或展示内容。
- 本次不改对话发送流程、消息流、知识库检索注入或产出物通知。
- 本次不改知识库清单规则、页面扫描规则、忽略规则或图谱构建脚本。
- 工作台外壳应从“亲自编排每个图谱阅读动作”变成“连接活地图阅读流程、图谱画布、右抽屉和对话入口”。
- 活地图阅读流程应拥有一个集中的工作台侧入口，负责把用户动作转成下一步抽屉状态、图谱命令和对话入口动作。
- 图谱画布继续作为图谱引擎适配层，负责接收图谱命令、回报选择变化、回报可见状态变化、打开页面请求和视图重置。
- 右抽屉继续作为展示层，负责显示当前抽屉状态并把按钮动作回传给活地图阅读流程。
- 选区提问和社区提问仍使用现有的提问内容生成规则，不在本次改变提示词结构或发送时机。
- 社区进入路径仍保留现有的抽屉退场和减少动态效果行为。
- 图谱刷新后的状态保留、隐藏提示、缺失节点清理和缺失社区处理仍使用现有语义。
- 允许新增一个工作台侧流程模块或同等抽象，但它必须服务于“活地图阅读流程”这个边界，而不是泛化成全应用状态管理。
- 若实现中发现必须改变用户可见行为才能完成收拢，应停止并把该变化拆到后续“用户可见体验改善”事项。

## Testing Decisions

- 好测试应该证明用户能观察到的活地图阅读行为没有变化，而不是锁死内部函数名或文件结构。
- 最高测试入口应是工作台侧活地图阅读流程：输入图谱选择、右抽屉动作、可见状态变化、图谱刷新和提问动作，断言输出的抽屉状态、图谱命令和对话入口动作。
- 优先复用现有测试先例：图谱摘要动作测试、图谱数据刷新测试、社区进入测试、右抽屉交互测试、图谱选择提问测试。
- 如果新增测试入口，理想情况下只新增一个高层流程测试入口，用来覆盖完整阅读路径，而不是给每个小 helper 重新铺一套测试。
- 必须覆盖节点选择、社区选择、手动多选、打开节点阅读、进入社区、返回社区摘要、找邻居、预览节点、固定节点、清理临时对象。
- 必须覆盖选区提问、社区提问、节点阅读提问，以及“在新对话中提问”的分支。
- 必须覆盖图谱刷新后的保留和清理：节点仍存在、节点消失、节点离开当前社区、当前阅读节点被筛选隐藏。
- 必须覆盖关闭右抽屉和清除选择：图谱相关抽屉会关闭，非图谱抽屉不被误关。
- 必须保留现有右抽屉渲染测试，证明展示结构和文案没有被内部收拢改掉。
- 必须运行前端相关测试；若改动触及共享图谱引擎类型边界，再运行图谱引擎测试。
- 完成后应至少做一次浏览器手动验收：打开工作台活地图，点节点，点社区，进入社区，打开节点阅读，从选区发问，确认界面行为没有明显变化。

## Out of Scope

- 不做用户可见的图谱阅读体验改善；该事项已记录为后续，排在本次内部收拢之后。
- 不改变活地图的视觉风格、关系线、节点样式、社区布局、镜头动画或图谱算法。
- 不改变右抽屉的外观、文案、按钮层级、宽度规则或全屏规则。
- 不改变对话发送流程、流式回复、工具状态、知识库检索或产出物通知。
- 不改变知识库清单规则、文件扫描规则、页面类型规则、忽略规则或缓存规则。
- 不新增 npm 依赖。
- 不做 Tauri、桌面打包、云同步或浏览器扩展相关工作。
